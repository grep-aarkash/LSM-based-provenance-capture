Review #81A
===========================================================================

Overall merit
-------------
1. Reject

Paper summary
-------------
There are provenance based security modules available for the Linux kernel which 
can be used for information flow tracking. Linux Security Modules also provide kernel hooks for implementing
arbitrary access control policies on system call invocation. While these two mechanisms provide similar
capabilities, they are not a 100% interoperable , i.e. the information captured by one might neither be exactly 
alike that captured by the other, nor would it be "complete". 
This work studies CamFlow - an information flow provenance system that uses the LSM hooks as the underlying 
capturing mechanism. CamFlow was written a while back, and there have been new releases of the kernel since then.
This work examines if the information captured by LSM is complete for a newer kernel, and if there exists 
any security related flow that it does not capture. In the ideal scenario, all security related flows get 
captured by LSM, show up on the CamFlow provenance graph.

Comments for author
-------------------
There are a lot of typos and grammatical errors; don't forget to use a grammar check. Also, the storyline is lacking - what fairy tale are you telling here? It would have been valuable to spend more time explaining how all these components fit in together and what is the workflow to analyze information flow using CamFlow. Also, the formatting in this paper was troubling. For example, tables without captions,  **huge** font size, and the interweaving use of the third person and first person in the narrative - the section on Anomaly Detection breaks the narrative and is quite odd. Also, the paper title does not need a question mark at the end.

Review #81B

Overall merit
-------------
1. Reject

Paper summary
-------------
This paper explores if the current LSM (Linux Security Modules) interface in CamFlow captures all the security-related flows in an updated kernel version of Linux. This is done to understand and evaluate if the current LSM interface is complete with respect to all security-related workflows. I was confused by the abstract :

-> "in the updated kernel version". Updated version of which kernel? Is there a specific reason why this kernel matters?

-> Is the goal to find violations of the mediation property of CamFlow? If so then why does it matter?

-> The LSM abbreviation is used before it was introduced. (I initially thought the paper was regarding LSM trees1)

Figure 1 is never explained or referenced

Review #81C
===========================================================================

Overall merit
-------------
2. Weak reject

Paper summary
-------------
There is existing work in data flow provenance in the Linux system, mostly from work by Pasquier and Georget. This work seems to involve the Linux Security Model, which is a way to place hooks on system calls. By placing hooks on system calls that cause information flows, we can track these flows as they occur. Moreover, there's a way to build a graph of these data flows, and analyze them to detect security violations. This paper raises the following questions: are the existing hooks sufficient to capture all flows, and can we show a formal mapping between the set of flows and the flow graph? The paper is incomplete so the questions are left unattended to for now.

Comments for author
-------------------
Please spend more time explaining what the different critical components are: LSM, Camflow, Graph Anomaly, etc. Please walk through the properties you wish to prove in detail. Please state clearly what it is you wish to prove by the end of the paper.

Review #81D
===========================================================================

Overall merit
-------------
2. Weak reject

Paper summary
-------------
Camflow showed that information flow security can be achieved with Linux Security Modules (LSM) if and only if all information flows are tracked by CamFlow (the completeness property), which was developed for Linux 4.3.  This paper follows up that work and asks two questions: (1) Does CamFlow still track all information flows in Linux 5? and (2) If #1 holds, will data leaks or intrusions show up as an anomaly in CamFlow's provenance graph?

The paper finds that for #1 there are some LSM hooks that are un-used by CamFlow and there are some system calls that trigger no hooks, and claims that attacks are possible because of these gaps. For #2, the paper finds that so long as the flow is being recorded, it is reflected in the provenance graph, but does not say if leaks or intrusions will show up as an anomaly.

Comments for author
-------------------
This paper would be much stronger if it explained the analysis performed and results in more detail, and put the findings in greater context. The paper explains the scripts that were run, but not how the analysis was done to find missing hooks or low coverage, e.g, was it done with manual inspection of code, or a comparison of script output? For the results, how many hooks were ignored in total? What does it mean that coverage is low? Why are the system calls that trigger no hooks important? 

For instance, the paper says "we can see that some of the hooks listed in Table 1 are ignored", but what does ignoring mean in this case? I'm assuming this means CamFlow does not use this LSM hook, but I am not sure. It would be great to discuss why CamFlow ignores these hooks? Are they unneeded? Are they recently added but CamFlow was not aware? Is there a significant risk by leaving these hooks out? Would it be easy or hard to add this hook to CamFlow?

What does it mean that "for __x64__sys_open the coverage is 8 out of 12"? Based on the background in the paper, my understanding of this was that a syscall was either instrumented or not.

- It would be great to define what an anomaly is and how they could be detected in the final provenance graph. The conclusion to the second question seems unclear, but to me it seems the answer is yes: so long as information flow is recorded, it shows up in the graph. Is there more to it than that?

Small nits:
- I'm not clear on where or how GraphChi was used to analyze the graph.
- Table 2 was unreferenced in the text.

Review #81F
===========================================================================

Overall merit
-------------
2. Weak reject

Paper summary
-------------
This work studies state of the provenance mechanism in Linux. Authors mostly focus on the using provenance for detecting security violation in Linux. After brief explanation of provenance as a relationship between entities, activities, and agents, authors describe current implementation in Linux that rely on Linux Security Modules (LSM). In a high level, authors want to answer two questions with this study: (1) is LSM interface provided by CamFlow portable, and (2) does security violation always show up as an anomaly in provenance graph. By portability authors intend to evaluate if CamFlow interface developed for Linux kernel 4.x suffices for Linux kernel 5.x. The work includes authors' approach to answer these questions and report on their initial findings. However, conclusive evidences are left out as a future work.

Comments for author
-------------------
I have already outlined my main concerns and below are minor concerns. There are mostly about improving the writing. Authors mix "my" and "our" work. Choose one and stick to it. I'd use plural.

Fig. 1 is not referenced nor explained.

Is this Latex from any Systems conference? Table captions look like a footnote. Page numbers are on top?

There are no space between brackets and a word. Similarly no space needed between quotes and words.
- ( or disprove )
+ (or disprove)

- ” Making sure that the system is following the correct protocol ”
+ ”Making sure that the system is following the correct protocol”

Make consistent spelling of "et al." and "i.e.," and "e.g.," and "CamFlow".

Full stops in the end of the sentence come after citation:
- [...] lack complete security coverage. [5]. 
+ [...] lack complete security coverage [5]. 

Too much use of "formal" and "formalism", sometimes redundant and sometimes unclear. Here are one examples for each case:
- The simplest way to define provenance is a formal set of documents [...]
+ The simplest way to define provenance is a set of documents [...]

- The first part of the challenge includes answering the question if the completeness formalism is enough to detect all security related flows.
+ The first part of the challenge includes answering the question if the definition of completeness is enough to detect all security related flows.

Note that "completeness formalism" above is replaced with "definition of completeness". Rule of thumb I suggest: if sentence reads valid without "formal" or "formalism" terms you may as well drop them. Adding those terms does not make your statement stronger.


Drop "Operating" word here, PASS and Hi-Fi are not operating systems:
- However, none of the above mentioned Operating systems provenance capture systems utilize LSM [...]
- However, none of the above mentioned systems provenance capture systems utilize LSM [...]

Discussion Topics come before Conclusion.

The 508 is "taught" by Margo not "taken".

There are many other minor nits on use of question marks, commas, and dots, which could be resolved with careful proofreading.
